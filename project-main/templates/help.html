Understanding code metrics and how Pridictx analyzes your code

Basic Code Metrics

Lines of Code (LOC)

Measures the total number of executable lines in a program.

LOC = Total Lines - (Comments + Blank Lines)
Higher LOC often indicates increased complexity and potential for bugs. Target maintaining smaller, focused modules.
Cyclomatic Complexity v(g)

Measures the number of linearly independent paths through program's source code.

v(g) = E - N + 2P where: E = Number of edges N = Number of nodes P = Number of connected components
Higher values indicate complex control flow and harder-to-test code. Aim for v(g) ≤ 10 per function.

Halstead Metrics

Program Vocabulary (n)
n = n1 + n2 where: n1 = unique operators n2 = unique operands
A larger vocabulary indicates more complex logic and potentially harder maintenance.
Program Volume (V)
V = N × log2(n) where: N = total operators + operands n = vocabulary size
Volume measures the information content of the program. Lower is generally better.
Program Difficulty (D)
D = (n1/2) × (N2/n2) where: n1 = unique operators N2 = total operands n2 = unique operands
Higher difficulty suggests more error-prone code and increased maintenance effort.
Time to Implement (T)
T = E / 18 seconds where: E = effort
Estimates the time needed to implement the program. Useful for project planning.
Number of Bugs (B)
B = V / 3000 where: V = volume
Theoretical estimation of the number of bugs in the implementation.

Line Count Metrics

Lines of Code

Total lines containing executable code

Key indicator of module size and potential complexity. Aim for smaller, focused modules.
Lines of Comments

Total lines containing only comments

Indicates documentation level. Focus on quality over quantity.
Lines of Code and Comments

Lines containing both code and comments

Helps identify inline documentation patterns. Prefer clear code over excessive comments.

Operator/Operand Metrics

Unique Operators (n1)

Count of distinct operators in the program

Reflects the variety of operations. Too many might indicate overcomplicated logic.
Unique Operands (n2)

Count of distinct operands in the program

Indicates variable and constant usage. High counts might suggest need for refactoring.

Control Flow

Branch Count

Total number of decision points in the code

Branch Count = if statements + loops + switch cases + logical operators
Higher branch counts indicate more complex decision paths. Consider breaking down complex methods.

Limitations

Predict

These models generally predict the likelihood of defects in certain parts of the code based on historical patterns, but they do not directly analyze the specific nature of the defect (e.g., logical errors or miscalculations).

Logical bugs

Logical bugs (e.g., errors in how a function or algorithm is implemented) are more difficult to predict using these datasets because they require deeper analysis of the code's logic, which is not typically captured by basic software metrics like lines of code or complexity.
